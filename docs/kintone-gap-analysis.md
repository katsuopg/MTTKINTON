## 重要な問題点

### 1. 権限システムの問題

| 問題 | 影響度 | 詳細 |
|------|--------|------|
| アプリ権限のOR集約 | **高** | 権限の剥奪ができない。everyoneでcan_view=trueなら、特定ユーザーでfalseにしても無効 |
| レコード権限の適用漏れ | **高** | 設定UIと評価関数はあるが、固定テーブルAPI（employees/customers等）で未適用 |
| フィールド権限の適用漏れ | **高** | `filterFieldsByPermission`の呼び出しが`employees/[id]`の1か所のみ |
| RLSが全テーブルで`USING(TRUE)` | **高** | API権限チェックのみに依存。直接Supabaseアクセスには無防備 |
| 権限チェック未設定API 19ルート | **中** | うち高リスク3件: process-action, employees/upload, kintone/download |
| 下位組織継承 | **低** | DBカラムのみ、ロジック未実装 |

### 2. プロセス管理の問題

| 問題 | 影響度 | 詳細 |
|------|--------|------|
| 作業者チェックなし | **高** | 認証済みユーザーなら誰でもアクション実行可能 |
| 作業者候補設定UIなし | **高** | process_status_assigneesテーブルは存在するが設定手段なし |
| record_assigneesへの割り当てなし | **高** | アクション実行後に新しい作業者を設定するロジックなし |
| アクション条件の評価ロジックなし | **中** | DB保存のみ、実行時に条件評価されない |
| マイグレーションファイル欠落 | **中** | プロセス管理8テーブルのマイグレーションSQLがローカルにない |

### 3. 同時編集制御なし

- 楽観ロック（etag/version/updated_atチェック）が未実装
- 同じレコードを複数ユーザーが同時編集した場合、後勝ちになる

### 4. マイグレーション欠落

以下のテーブルがコード上使用されているがマイグレーションファイルが見当たらない:
- `app_notification_rules`
- `app_record_history`
- プロセス管理8テーブル (`process_definitions`, `process_statuses`, `process_actions` 等)

## 良好な実装

- **フォームビルダー**: 28フィールドタイプ、D&D配置、設定UIが包括的
- **通知エンジン**: 5トリガー、5種類の通知先、テンプレート展開
- **Webhookエンジン**: fire-and-forget方式で各種イベントに対応
- **UIの統一性**: CLAUDE.md規約にほぼ準拠（tableStyles/detailStyles/AppListToolbar/DetailPageHeader）
- **多言語対応**: ja/en/thの3言語が全ページで対応
- **モバイル対応**: レスポンシブ+ボトムナビゲーション+サイドバー切替
- **動的アプリフレームワーク**: Kintone互換のアプリ作成/テンプレート/コピー/削除/復旧が一通り実装
